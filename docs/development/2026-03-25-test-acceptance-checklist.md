## 3. 验收状态标记规则

每项用例统一按下面 3 种状态记录：

- `正式上线可测`
  - 代表可用正式后端链路验收
- `演示可测`
  - 代表页面和流程可跑，但底层仍依赖 mock / fallback / 本地状态
- `暂不可测`
  - 代表当前还没有形成有效测试能力

## 5. 企业端测试清单

当前建议状态：`演示可测`

### 5.1 注册与入驻

通过条件：

- 企业端可注册
- 登录后能进入企业端
- 入驻页为单列步骤式
- 合作偏好为结构化选项
- 资料可直接上传或稍后补交
- 提交后进入企业工作台

### 5.2 发布任务

通过条件：

- 能输入任务标题、需求描述、预算
- 不选模板也能继续
- AI 拆解能正常生成
- AI 标签更偏业务 / 交付标签
- 推荐工具优先国内可用工具
- 发布成功后自动弹出推荐人才

### 5.3 人才广场

通过条件：

- 企业端人才广场以办公端列表页承接，不再退回营销式大卡片瀑布
- 顶部可看到轻量总览和快速筛选
- 主体优先为行式列表或表格，支持行级操作
- 右侧详情区在切换行时保持稳定承接，不与主列表争抢滚动心智
- 每行人才至少能看到姓名、方向、评分、响应速度、能力摘要和行级操作
- 可从行级动作进入人才详情、继续筛选，或直接去发布任务
- 进入人才详情后返回人才广场时，筛选状态应能保留

### 5.4 选人与聊天

通过条件：

- 推荐人才可直接选择
- 也可跳过后续再选
- 选中人才后自动进入正确聊天房间
- 聊天列表主标题显示任务名称
- 最近更新的房间排最前
- 从工作台、通知中心、审批中心、记录详情进入聊天时，聊天页能展示轻量来源提示与返回来源入口
- 刷新聊天页或复制当前 URL 直达时，`taskId / room / recordId / source` 仍能保持一致，不会被旧请求回滚到其它房间
- 从企业工作台和记录详情进入聊天时，URL 能保留 `taskId / room / recordId / source`
- 从协作空间进入聊天时，若 URL 带了 `nodeId`，聊天页要保留 `taskId / room / nodeId / source=workspace`
- 当聊天页缺少 `room` 但保留 `taskId` 时，应优先回到当前任务对应的房间，而不是回到最近会话
- 在聊天页手动切换到其它房间后，旧 `nodeId / source` 不应继续粘在新房间上
- 从记录详情进入聊天时，右侧上下文应显示当前记录的阶段、金额、评级、最近留痕和“查看记录”入口
- 从聊天页点击“查看记录”或“返回来源”时，应回到原 `recordId` 对应的记录详情，并保留 `tab`

### 5.5 任务确认与变更

通过条件：

- 企业可发送任务确认卡
- 人才未处理前，企业不能继续发送第二个变更
- 企业可以撤回当前待确认变更
- 企业可修改说明、工期、预算后重新发送
- 历史显示版本和处理记录

### 5.6 协作与验收

通过条件：

- 企业协作空间可在多个任务之间切换
- 企业可查看节点、人才提交、AI 审核、企业反馈
- 企业可发起提前完成
- 企业可发起取消任务

- 从企业工作台、聊天、记录列表、记录详情进入协作空间时，能落到正确任务而不是最近任务

- 刷新协作空间或复制当前 URL 直达时，`taskId / room / nodeId / source` 仍能回到正确任务和正确节点

### 5.7 PC 待办 / 通知 / 审批入口

- 人才工作台顶部能看到统一聚合的高优先级待办入口

- 工作台顶部待办入口优先消费 `notificationItems / notificationGroups`；删除这两个字段后，才回退到旧的 `attentionItems`
- 工作台顶部待办入口的展开区只展示“补充概览”，不应继续用“查看全部待办”误导用户
- 下一阶段收口后，工作台顶部待办入口、模块弹层入口和首页快捷入口只允许进入通知中心 / 审批中心，不应再从首页直接跳聊天 / 协作空间 / 记录详情 / 验收页
- 企业端入口优先落到审批式处理页面或对应上下文
- 人才端入口优先落到确认式处理页面或对应上下文
- 企业端和人才端都能进入独立通知中心页
- 通知中心页能按分组、列表和右侧摘要承接高优先级事项
- `GET /api/business` 与 `GET /api/talent` 返回 `notificationItems / notificationGroups` 时，通知中心优先展示真实通知项；删除这两个字段后，页面仍能回退到旧的 `attentionItems`
- 从企业工作台进入审批中心、从人才工作台进入通知中心时，若 URL 带 `group`，左侧分组要默认落到对应类别
- 从企业工作台进入审批中心、从人才工作台进入通知中心时，若 URL 再带 `itemId / taskId / room / recordId`，中心页要直接选中对应事项，而不是只停在该分组第一条
- 首页透传的 `group + itemId / taskId / room / recordId` 当前只用于中心页预选，不再作为首页对象页直达参数
- 企业端顶部共享待办条目当前应统一先进入审批中心，不能再直接跳聊天、协作空间、记录详情或验收页
- 人才端顶部共享待办条目当前应统一先进入通知中心，不能再直接跳聊天、协作空间、记录详情或验收页
- 从企业端 / 人才端工作台模块弹层进入中心页时，应按模块语义落到更接近当前模块的事项，而不是一律落到第一个待办
- 首页模块弹层的 secondary 动作只允许保留发布任务、任务广场这类泛入口，不应继续保留聊天、协作空间、记录详情对象页直达
- 首页相关验证当前应统一按“首页 -> 中心页 -> 对象页”检查，而不是继续按“首页 -> 对象页”检查

- 通知中心右侧当前对象动作带进入聊天、协作空间、记录详情、验收页后，返回仍要恢复原分组与原事项
- 审批中心右侧当前对象动作带进入聊天、协作空间、记录详情、验收页后，返回仍要恢复原分组与原事项
- 右侧当前对象动作带的摘要高亮应优先跟随当前选中对象，而不是继续跟着列表前几项漂移
- 通知中心 / 审批中心右侧动作带当前应统一消费共享对象动作合同；只有共享合同缺失时，才允许回退 `route / secondaryRoute`
- 共享对象动作合同生成的所有动作都必须继续保留 `group + itemId + taskId + room + recordId`
- 当中心页 URL 携带的 `itemId / taskId / room / recordId` 对应事项已失效时，页面必须进入显式失效态，不能静默跳到无关首项
- 中心页失效态下应继续保留原 `group` 与来源语义；用户手动切组或手动重新选择事项后，失效态才允许被清掉
- 中心页失效态下，右侧只允许展示安全恢复动作，不能继续保留会把用户带去无关对象的常规主动作

- 中心页失效态下，顶部摘要卡的 primary / secondary 动作也必须切成恢复动作，不能继续保留“去处理 / 去协作空间 / 去通知中心 / 去审批中心”这类常规对象动作
- 中心页失效态下，底部 footer 动作也必须和顶部摘要卡、右侧空态使用同一组恢复动作；三处按钮名称与目标不能互相冲突
- 从通知中心进入聊天、协作空间、记录详情后，页面来源能识别为 `notifications`
- 从通知中心进入聊天时，若 URL 带 `group + itemId + taskId / room / recordId`，聊天页返回来源后要恢复原通知分组与原事项
- 从通知中心进入协作空间或记录详情后，点击“返回来源”时，要回到通知中心原分组与原事项

- 记录详情、协作空间、验收页的返回按钮文案应贴合真实目标；例如从聊天进入记录详情后应显示“返回聊天”，从记录详情进入协作空间再进入验收后，返回按钮应先后显示“返回协作空间”“返回记录详情”
- 从中心页失效态执行“查看当前分组首项”或“清除当前定位”后，再进入记录详情 / 协作空间 / 验收页时，页面必须绑定恢复后的新对象或新上下文，不能继续携带旧失效锚点

- 审批动作返回后，列表、摘要卡与右侧详情都应直接刷新到最新 `approvalItems / approvalGroups`
- 当审批动作返回 `actionBlocked` 时，页面必须展示阻塞原因，但仍按返回的新列表刷新，不能误判成已完成
- 从审批中心进入协作空间、聊天、记录详情后，页面来源能识别为 `approvals`
- 从审批中心进入聊天、协作空间或记录详情后，点击“返回来源”时，要回到审批中心原分组与原事项
- 从审批中心进入记录详情后再进入协作空间时，返回应按“协作空间 -> 记录详情 -> 审批中心原分组与原事项”的顺序恢复

- AcceptancePage 当前应继续保留 `itemId / group / taskId / recordId / room / source / originSource`
- 若原 `itemId` 已失效，页面才允许按 `recordId -> taskId -> group` 的顺序回退，不能静默跳到无关首项
- 从记录详情或协作空间点击“返回来源”时，若来源为审批中心，应能返回审批中心
- 从工作台进入聊天、记录、协作空间时，URL 仍能保留 `taskId / room / recordId / source`
