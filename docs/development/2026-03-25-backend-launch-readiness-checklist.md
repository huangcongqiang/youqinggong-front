# 后端上线前能力清单

更新时间：2026-03-25

这份文档只回答一个问题：当前后端距离“可上线承载真实用户”还差什么，应该按什么顺序补。

## 1. 文档目标与范围

这份清单是后端上线判断的事实源，用来回答：

- 后端现在能不能直接上线
- 上线还差哪些阻塞项
- 下一批后端开发优先做什么
- 哪些能力只是联调可用，不等于生产可用

不回答的内容：

- 详细页面设计
- 详细接口字段表
- 详细部署命令

这些内容分别以其他主文档为准：

- 技术实现边界：[技术文档](../technical/2026-03-21-ai-talent-market-technical.md)
- 接口字段与示例：[接口文档](../technical/2026-03-21-api-contract.md)
- 开发现状与联调说明：[开发文档](./2026-03-21-ai-talent-market-development.md)
- 部署与回滚：[部署文档](./2026-03-21-deployment-guide.md)

## 2. 当前架构现状

当前项目后端分成两层：

- `backend/src/main/java`
  - root mock API
  - 用途：本地联调、演示、小规模多人测试
- `backend/spring-app`
  - Spring Boot + JPA + MySQL 正式服务骨架
  - 用途：继续演进为真实线上服务

当前结论：

- **现在不能把 root mock API 当作正式线上后端。**
- **真正的上线路径必须以 `spring-app + MySQL` 为主。**
- root mock API 只能继续承担：
  - 页面联调
  - 产品流程验证
  - 小范围演示

### 2.1 root mock API 现状

已实现：

- 官网、企业端、人才端、后台管理的本地联调接口
- 轻量登录 / 注册 / `me`
- 任务发布、选人、协商、进度、验收、评分
- 提前完成
- 双向取消任务
- 腾讯 IM 配置下发与部分回调预留
- 本地状态持久化
- 基础并发保护

边界：

- 通过单仓储串行访问保证状态安全
- 通过本地 `.local/*.json` 文件保存状态
- 偏向“联调稳定”，不适合真正高并发

### 2.2 spring-app 现状

已实现：

- Spring Boot 3 + JPA + MySQL 工程骨架
- 入驻、后台管理、任务生命周期实体与仓储基础
- 公共展示和部分协作读模型基础
- 聊天 / IM 相关实体与仓储基础

边界：

- 还没有正式认证鉴权
- 还没有完整任务主写链
- 还没有完整聊天写链和回写链
- 仍存在 fallback / demo 读模型
- 还没有正式实时能力和上传能力

## 3. 上线阻塞项总览

### 3.1 P0 阻塞项

这些能力只要缺任意一项，都不建议正式开放给真实用户：

- 正式认证鉴权
- 任务主写链迁移到 `spring-app + MySQL`
- 聊天 / 腾讯 IM 写链与回写链
- 文件上传与附件存储
- 至少一条正式实时状态同步链路
- 基础监控、日志、健康检查、回滚路径

### 3.2 P1 高优先级补强项

- 数据库事务边界和幂等控制
- 关键查询索引、分页和慢查询治理
- 错误告警与日志检索
- 压测基线
- 限流、防刷和关键写接口保护

### 3.3 P2 二阶段优化项

- 缓存层
- 更细的 IM 附件能力
- 更完整的事件总线
- 更细粒度的数据分析和审计视图

## 4. 认证与权限

### 当前状态

- 前端已经有登录页和轻量登录态
- root mock API 已有本地联调用的登录 / 注册 / `me`
- `spring-app` 还没有正式认证链

### 已实现

- mock token
- 本地持久化会话
- 企业端 / 人才端基础角色区分

### 未实现

- 正式登录 / 注册 / 登出
- 密码哈希与正式凭据存储
- JWT 或 Session 方案
- 角色权限边界
- 当前用户上下文注入
- 正式 `GET /api/auth/me`

### 上线前必须补齐

- 认证接口落到 `spring-app`
- 密码不再以 mock 方式处理
- 所有关键写接口不再信任前端裸传 `userId`
- 前端登录态切到正式服务

### 验收标准

- 未登录用户无法访问企业端 / 人才端 / 管理后台的受保护接口
- 企业、人才、管理员权限边界清晰
- 登录态可过期、可退出、可追踪

## 5. 任务主链

### 当前状态

- 任务流程在 root mock API 里基本可走通
- `spring-app` 有任务实体和部分生命周期基础

### 已实现

- 发布任务
- AI 拆解确认
- 选人
- 任务确认与变更版本
- 撤回变更
- 进度提交
- 提前完成
- 双向取消任务
- 验收
- 双向评分

### 未实现

- 上述写链尚未完整迁移到 `spring-app`
- 事务边界未正式落库验证
- 当前流程仍有 fallback / mock 路径

### 上线前必须补齐

- 以下写路径全部由 `spring-app + MySQL` 承接：
  - 发布任务
  - AI 确认
  - 选人
  - 任务确认 / 修改 / 撤回
  - 进度提交
  - 提前完成
  - 取消任务
  - 验收
  - 评分
- 预算字段进入正式数据库模型
- 状态流转可追踪、可审计

### 验收标准

- 双角色完整主流程在正式后端跑通
- 重复点击、并发点击不会造成重复写入或脏状态
- 状态变更可回溯到操作者和时间

## 6. 聊天与腾讯 IM

### 当前状态

- 聊天页面已具备联调能力
- 文本即时消息优先走腾讯 IM
- 附件和摘要仍有平台侧 fallback
- root mock API 承担了大量会话业务对象拼装

### 已实现

- 会话初始化
- 房间定位
- 文本消息基础收发
- 腾讯 IM 配置下发
- 部分回调预留

### 未实现

- 正式用户映射
- 正式房间映射
- IM 回调验签
- 消息持久化主链
- 房间摘要回写主链
- 任务状态联动主链
- 附件消息完整线上化
- 异常重试与幂等

### 上线前必须补齐

- 会话列表、房间详情、首次发起聊天、发消息全部由正式服务承接
- 腾讯 IM 回调入库并完成业务回写
- 附件消息纳入正式文件链路
- 房间摘要和任务状态保持一致

### 验收标准

- 用户从人才详情、任务详情、工作台发起聊天都能进入正确房间
- 文本、图片、文件消息都能完整收发并回写
- 腾讯 IM 回调异常时有可重试与可追踪机制

## 7. 文件上传与附件存储

### 当前状态

- 当前更多是“附件元信息 + 页面预览”
- 不是正式文件服务

### 已实现

- 聊天页选择图片 / 文件
- 协作空间提交附件元信息
- 入驻流程记录资料文件名

### 未实现

- 对象存储
- 正式上传鉴权
- 文件索引表
- 预览策略
- 删除 / 失效策略
- 附件权限控制

### 上线前必须补齐

- 上传签名或上传中转接口
- 对象存储
- 附件索引与业务关联
- 入驻、聊天、进度、作品附件统一治理

### 验收标准

- 文件可上传、可回看、可校验、可审计
- 无权限用户无法查看受保护附件
- 失效 / 删除策略可执行

## 8. 实时能力

### 当前状态

- 当前更准确的状态是“IM 即时消息 + 关键字段轮询刷新”
- 这不等于正式实时架构

### 已实现

- 聊天即时消息基础能力
- 关键业务字段准实时刷新

### 未实现

- 正式 WebSocket / SSE 业务事件推送
- 任务状态实时同步主链
- 协作节点实时刷新主链

### 上线前必须补齐

- 至少一种正式实时机制：
  - WebSocket
  - SSE
- 至少覆盖以下关键事件：
  - 任务确认
  - 提前完成
  - 取消任务
  - 进度提交
  - 验收

### 验收标准

- 关键业务状态不再主要依赖前端轮询
- 双端在关键节点能及时看到一致状态

## 9. 并发承载与性能

### 当前状态

- root mock API 通过串行保护提升了稳定性
- 这不等于支持高并发

### 已实现

- 基础并发保护
- 本地状态原子写
- 前端部分路由懒加载
- 聊天页自适应刷新

### 未实现

- 数据库事务边界正式验证
- 索引策略
- 分页规范
- 慢查询治理
- 限流
- 压测基线
- 高并发场景下的读写隔离和削峰方案

### 上线前必须补齐

- 关键写操作事务边界清晰
- 关键读查询有索引和分页
- 慢查询治理和限流策略可落地
- 有基础压测结果

### 验收标准

- 并发发布任务、并发选人、并发聊天、并发改任务不会出现状态覆盖
- 高峰期关键接口响应时间在可接受范围

## 10. 数据一致性与事务

### 当前状态

- root mock API 依赖内存对象和本地文件
- `spring-app` 事务边界尚未形成正式规范

### 上线前必须补齐

- 任务状态流转事务边界
- 聊天回写与业务状态更新边界
- IM 回调幂等
- 附件上传与业务对象绑定边界
- 提前完成 / 取消任务 / 验收等多方动作的一致性规则

### 验收标准

- 同一个任务不会出现“双确认”“双取消”“重复验收”这类脏状态
- 同一事件重复回调不会导致重复结算或重复状态迁移

## 11. 监控、日志与告警

### 当前状态

- 当前主要是本地日志和手工排查

### 上线前必须补齐

- 结构化日志
- 错误日志检索
- 健康检查
- 核心指标监控
- 告警机制

建议至少覆盖：

- 登录失败率
- 任务发布失败率
- IM 回调异常率
- 上传失败率
- 关键接口耗时

### 验收标准

- 出现错误时能追到请求、用户、任务、房间
- 上线后关键异常可被主动告警而不是被动发现

## 12. 安全与配置管理

### 当前状态

- 本地联调已支持 `.env.local`
- 但生产配置和密钥治理仍需正式化

### 上线前必须补齐

- 生产密钥脱离代码仓库
- 腾讯 IM、DeepSeek、数据库、对象存储密钥集中管理
- 回调验签
- 基础防刷、限流
- 关键输入校验

### 验收标准

- 不再依赖本地开发默认值
- 敏感配置可轮换、可审计

## 13. 测试与发布

### 当前状态

- 当前仍以手工联调和演示验证为主

### 上线前必须补齐

- 双角色主流程回归测试
- 聊天 / 上传 / 验收闭环测试
- 错误路径测试
- 回滚演练
- 数据库初始化与迁移验证

### 验收标准

- 企业端、人才端、后台管理的关键主流程都有可复用测试清单
- 发布失败时有明确回滚路径

## 14. 推荐迁移顺序

建议按下面顺序推进，而不是平铺并行：

1. 认证鉴权
2. 任务主写链
3. 聊天 / IM 回写链
4. 文件上传
5. 实时同步
6. 缓存 / 监控 / 限流 / 压测

原因：

- 没有认证，上下文不可信
- 没有任务主链，聊天没有业务归属
- 没有聊天回写和上传，协作体验不完整
- 没有实时与监控，上线后体验和排障都会很被动

## 15. 第一阶段切换原则

第一阶段不要先大改前端，而是让 `spring-app` 兼容现有 `/api/*` 返回结构。

目标：

- 前端尽量不大改即可切后端
- 先把真实主链切到正式服务
- 再逐步清理 fallback 字段和演示字段

建议第一批接管范围：

- 登录 / 注册 / `me`
- 发布任务
- AI 确认
- 选人
- 任务确认 / 修改 / 撤回
- 进度提交
- 提前完成
- 取消任务
- 验收
- 评分

建议第二批接管范围：

- 会话列表
- 房间详情
- 发消息
- 聊天初始化
- IM 回调落库
- 房间摘要同步

## 16. 上线前通过条件

上线前至少满足以下条件：

- [ ] `spring-app` 已提供正式登录 / 注册 / me
- [ ] 任务主写链全部由 `spring-app + MySQL` 承接
- [ ] 腾讯 IM 消息与附件回写正常
- [ ] 提前完成闭环正常
- [ ] 取消任务闭环正常
- [ ] 错误日志可追踪
- [ ] 健康检查可用
- [ ] 回滚路径明确
- [ ] 双角色完整主流程已跑通
- [ ] 关键接口压测已执行

## 17. 上线后第一阶段观察项

上线后一周重点盯这些指标：

- 登录失败率
- 任务发布成功率
- 选人成功率
- 提前完成触发率与审核通过率
- 取消任务率
- 腾讯 IM 回调失败率
- 上传失败率
- 关键接口耗时

同时重点观察：

- 是否存在重复状态迁移
- 是否存在房间错绑 / 消息错房间
- 是否存在附件无法回看
- 是否存在评分和结算比例不一致
