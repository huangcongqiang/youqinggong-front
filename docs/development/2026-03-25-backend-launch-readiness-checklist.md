# 后端上线前能力清单

更新时间：2026-03-30

这份文档只回答一个问题：当前后端距离“可上线承载真实用户”还差什么，应该按什么顺序补。

## 1. 文档目标与范围

这份清单是后端上线判断的事实源，用来回答：

- 后端现在能不能直接上线
- 上线还差哪些阻塞项
- 下一批后端开发优先做什么
- 哪些能力只是联调可用，不等于生产可用

不回答的内容：

- 详细页面设计
- 详细接口字段表
- 详细部署命令

这些内容分别以其他主文档为准：

- 接口字段与示例：[接口文档](../technical/2026-03-21-api-contract.md)

## 2. 当前架构现状

当前项目后端分成两层：

- `backend/src/main/java`
  - root mock API
  - 用途：本地联调、演示、小规模多人测试
- `backend/spring-app`
  - Spring Boot + JPA + MySQL 正式服务骨架
  - 用途：继续演进为真实线上服务

当前结论：

- **现在不能把 root mock API 当作正式线上后端。**
- **真正的上线路径必须以 `spring-app + MySQL` 为主。**
- **基于 `spring-app + MySQL` 的正式主链，当前已经通过主线 9 的最终验收，可标记为 `正式上线可测`。**
- root mock API 只能继续承担：
  - 页面联调
  - 产品流程验证
  - 小范围演示

### 2.1 root mock API 现状

已实现：

- 官网、企业端、人才端、后台管理的本地联调接口
- 轻量登录 / 注册 / `me`
- 任务发布、选人、协商、进度、验收、评分
- 提前完成
- 双向取消任务
- 腾讯 IM 配置下发与部分回调预留
- 本地状态持久化
- 基础并发保护

边界：

- 通过单仓储串行访问保证状态安全
- 通过本地 `.local/*.json` 文件保存状态
- 偏向“联调稳定”，不适合真正高并发

### 2.2 spring-app 现状

已实现：

- Spring Boot 3 + JPA + MySQL 工程骨架
- 入驻、后台管理、任务生命周期实体与仓储基础
- 公共展示和部分协作读模型基础
- 聊天 / IM 相关实体与仓储基础

边界：

- 已完成正式认证、正式主写链、正式聊天写链、正式协作 / 验收写模型、正式中心页读模型与双跑切换
- 已完成上传底座、IM callback 验签 / 附件回写第一批、SSE / readiness / health 最小可观测性与高风险接口保护
- 阶段七最终验收已完成：九组真实主路径、整端 UI 审查、对象页来源链 / 返回链与多跳聊天 / 协作路径回归都已过门
- 当前仍保留 root mock API 作为演示 / 联调 fallback，但它不再代表正式上线链路
- 已完成管理台合规读模型第一刀正式化：`/api/admin/compliance` 已移除空风险工单 demo fallback，并补第一批 `organizationPermissions`
- 已完成管理台读模型第二刀正式化：`/api/admin/dashboard` 与 `/api/admin/users` 已移除空列表 demo fallback，当前会返回正式空队列或真实投影
- 已完成管理台读模型第三刀正式化：`/api/admin/tasks` 已移除无任务场景下的整页 demo fallback，当前会返回正式空态与真实任务投影
- 已完成管理台前端与 fallback 对齐：后台合规页当前已正式消费 `organizationPermissions`
- 已完成高风险接口保护第一刀：`POST /api/auth/login` 当前已补最小登录失败限流，连续失败后返回正式 `429`
- 已完成高风险接口保护第二刀：`POST /api/uploads/presign` 当前已补最小 burst guard，突发申请上传会话达到阈值后返回正式 `429`
- 已完成高风险接口保护第三刀：`POST /api/tencent-im/callback/events` 与 `POST /api/tencent-im/callback/messages` 当前已补 callback header/secret 校验，未通过时返回正式 `401`；`POST /api/tasks/publish` 也已补 burst guard，超阈值时返回正式 `429`
- 已完成高风险接口保护第四刀：`GET /actuator/health` 与 `GET /actuator/health/readiness` 当前已补 `requestProtection` 组件，并冻结登录限流、上传 burst guard、发布 burst guard、腾讯 IM callback 保护四类开关；`businessRealtimeEvent` 当前也已补 `emitFailureThreshold / lastConnectAt / lastPublishAt / lastEmitFailureAt`，`uploadStorage` 当前已补 `checkedAt`，`tencentImConfig` 当前已补 `callbackProtectionEnabled`
- 已完成高风险接口保护第五刀：部署文档与上线清单当前已冻结最小运行手册、最小回滚路径与慢查询 / 索引关注点；本地最小验证命令已固定为 actuator / callback / presign guard 三条测试命令
- 主线 9 阶段七最终验收已完成：企业 PC、人才 PC、企业 H5、人才 H5、后台管理、`spring-app`、上传链、IM callback、`SSE / readiness / health` 九组真实主路径都已过门；本地浏览器终验使用 `spring-app` 的 H2 运行态验证正式链合同，部署目标仍以 `spring-app + MySQL` 为准

## 3. 上线阻塞项总览

### 3.1 P0 阻塞项

这些能力只要缺任意一项，都不建议正式开放给真实用户：

- 正式认证鉴权
- 任务主写链迁移到 `spring-app + MySQL`
- 聊天 / 腾讯 IM 写链与回写链
- 文件上传与附件存储
- 审批、请款、结算等关键业务闭环
- 至少一条正式实时状态同步链路
- 基础监控、日志、健康检查、回滚路径
- 财务审计、组织权限与争议第二阶段仲裁链
### 3.2 P1 高优先级补强项

- 数据库事务边界和幂等控制
- 关键查询索引、分页和慢查询治理
- 错误告警与日志检索
- 压测基线
- 限流、防刷和关键写接口保护（当前已完成 `/api/auth/login` 与 `/api/uploads/presign` 两刀）
- 组织权限体系与争议第二阶段仲裁 / 审计链

### 3.3 P2 二阶段优化项

- 缓存层
- 更细的 IM 附件能力
- 更完整的事件总线
- 更细粒度的数据分析和审计视图

## 4. 认证与权限

### 当前状态

- 前端已经有登录页和轻量登录态
- root mock API 已有本地联调用的登录 / 注册 / `me`
- `spring-app` 已完成第一批任务主链 actor 边界正式化，但完整认证链仍未完成

### 已实现

- mock token
- 本地持久化会话
- 企业端 / 人才端基础角色区分
- `spring-app` 当前已让任务发布、AI 拆解确认、选人、协商确认、闭环摘要、进度、验收、评分这批前台主链接口从 token 解当前 actor

### 未实现

- 正式登录 / 注册 / 登出全链切换到 `spring-app`
- 密码哈希与正式凭据存储
- JWT 或 Session 方案完整正式化
- 角色权限边界
- 全量接口当前用户上下文注入
- 正式 `GET /api/auth/me`

### 上线前必须补齐

- 认证接口落到 `spring-app`
- 密码不再以 mock 方式处理
- 所有关键写接口不再信任前端裸传 `userId`
- 前端登录态切到正式服务

### 验收标准

- 未登录用户无法访问企业端 / 人才端 / 管理后台的受保护接口
- 企业、人才、管理员权限边界清晰
- 登录态可过期、可退出、可追踪

### 未实现

### 上线前必须补齐

### 当前状态

- 当前实现态主要覆盖任务确认、进度、提前完成、取消任务、验收和评级
- 审批中心与请款最小闭环已进入 `spring-app`
- 开票与对账最小闭环已进入 `spring-app`
- `Settlement` 最小执行对象已进入 `spring-app`
- 对账异议当前已可生成 `Dispute` 与 `RiskTicket`，记录详情 / 闭环详情 / 后台合规页都可回读
- 完整财务审计、正式支付出账与争议第二阶段处理仍未形成完整正式后端主链

### 未实现

- 分账 / 结算出账
- 财务状态流转与审计
- 争议第二阶段处理、仲裁与风险升级联动

### 上线前必须补齐

- 审批、请款、开票、对账、结算实体与状态机进入 `spring-app + MySQL`
- `Dispute / RiskTicket` 第一阶段正式主链保持稳定，可继续向仲裁 / 复核扩展

- 金额、比例、评级、结算结果一致
- 财务操作具备审计留痕

### 验收标准

- 企业和人才都能看到正确的财务状态
- 不会出现重复请款、重复结算或金额不一致

## 10. 数据一致性与事务

### 当前状态

- root mock API 依赖内存对象和本地文件
- `spring-app` 事务边界尚未形成正式规范

### 上线前必须补齐

- 任务状态流转事务边界
- 聊天回写与业务状态更新边界
- IM 回调幂等
- 附件上传与业务对象绑定边界
- 提前完成 / 取消任务 / 验收等多方动作的一致性规则
- 请款状态、付款状态、评级结果与结算结果的一致性规则

### 验收标准

- 同一个任务不会出现“双确认”“双取消”“重复验收”这类脏状态
- 同一事件重复回调不会导致重复结算或重复状态迁移
- 同一任务不会出现评级结果、请款结果、付款结果彼此矛盾

## 11. 监控、日志与告警

### 当前状态

- 结构化日志
- 错误日志检索
- 健康检查
- 核心指标监控
- 告警机制

建议至少覆盖：

- 登录失败率
- 任务发布失败率
- IM 回调异常率
- 上传失败率

### 验收标准

- 上线后关键异常可被主动告警而不是被动发现

### 当前最小运行手册已冻结的检查点

当前最少要能回答这几个问题：

- 打到的是不是 `spring-app`，而不是静态前端或其它服务
- `requestProtection` 四类保护是否处于启用态
- `businessRealtimeEvent` 是否出现持续 emit failure
- 上传链当前是否可写，`checkedAt` 是否持续刷新
- 腾讯 IM callback 保护是否完整启用

本地最小验证命令当前冻结为：

## 14. 推荐迁移顺序

建议按下面顺序推进，而不是平铺并行：

1. 认证鉴权
2. 任务主写链
3. 聊天 / IM 回写链
4. 审批 / 请款 / 结算
5. 文件上传
6. 实时同步
7. 风控 / 组织权限 / 缓存 / 监控 / 限流 / 压测

原因：

- 没有认证，上下文不可信
- 没有任务主链，聊天没有业务归属
- 没有审批与结算，真实交易链不完整
- 没有聊天回写和上传，协作体验不完整
- 没有实时与监控，上线后体验和排障都会很被动

## 15. 第一阶段切换原则

第一阶段不要先大改前端，而是让 `spring-app` 兼容现有 `/api/*` 返回结构。

目标：

- 前端尽量不大改即可切后端
- 先把真实主链切到正式服务
- 再逐步清理 fallback 字段和演示字段

- 登录 / 注册 / `me`

- 选人
- 任务确认 / 修改 / 撤回
- 进度提交
- 提前完成
- 取消任务
- 验收
- 评分
- 发单记录 / 接单记录

建议第二批接管范围：

- 审批中心

- 会话列表
- 房间详情
