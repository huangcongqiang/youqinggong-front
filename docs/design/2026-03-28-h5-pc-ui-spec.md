# H5 仿 App 与 PC Web 界面规范

> 版本：2026-03-28  
> 事实源说明：本文件基于“规范文件夹/01-规范文档”整理，用于统一 H5 与 PC 的页面范式、导航结构与禁忌清单。

## 1. 规范目标

回答四件事：

- H5 仿 App 应该长什么样
- PC Web 应该长什么样
- 两端分别适合承载什么
- 哪些界面做法不允许出现

## 2. 总原则

- 以任务与协作为核心，而不是以页面展示为核心
- 高频路径优先，低频说明隐藏
- H5 优先单任务、单焦点、单列阅读
- PC 优先多任务、强信息密度、批量处理
- 清楚、稳定、可信优先于炫技

## 3. H5 规范

### 3.1 产品形态

H5 必须像标准移动 App，而不是缩窄后的网页。

标准外壳：

- 顶部轻量导航栏
- 底部 4 个一级 Tab
- 主体单列内容流
- 二级功能通过跳转、sheet、抽屉承接
- 底部安全区完整预留

### 3.2 页面范式

- 登录与注册：独立页面，单页单步
- 工作台：只显示待办、状态与主入口；首屏优先露出 2 到 3 个关键动作，其余入口下沉到二级层
- 列表页：单列卡片或单列列表
- 详情页：全屏承接单对象
- 聊天：会话列表页与聊天详情页分开
- 协作：以轻协作为主，不承接 PC 级复杂管理；首屏优先当前任务、关键节点、主动作

### 3.3 禁忌

- 不允许把 PC 双栏工作台直接缩到手机上
- 不允许聊天页同屏承载会话列表 + 聊天详情 + 大量辅助信息
- 不允许在首页、工作台、详情页堆叠长说明模块
- 不允许输入区被消息和任务卡挤出视口

## 4. PC 规范

### 4.1 产品形态

PC 承担复杂管理与办公效率。

推荐外壳：

- 左侧业务导航
- 顶部状态与筛选
- 中部主工作区
- 详情抽屉 / 右侧上下文栏
- 工作台优先表现为桌面驾驶舱，不再套官网顶栏心智

### 4.2 页面范式

- 工作台：驾驶舱而非展示页
- 列表页：默认按办公端列表页收口，以“顶部摘要 + 筛选区 + 行式列表/表格 + 详情抽屉 / 右侧上下文 + 批量动作”为主，不再以营销式卡片流承载核心管理场景
- 过渡态允许先以“摘要条 + 筛选区 + 行式列表 / 资源目录 + 固定详情承接”落地，但最终目标仍是更高密度的办公端列表与批量处理
- 工作台顶部的待办、通知与审批入口应统一聚合为共享入口层，企业端偏审批式处理，人才端偏确认式处理，不再散落在多个模块卡里各自独立出现
- 共享入口层优先承接高优先级事项：待确认、待修改、待评级、待取消、待回看，不要继续把这些事项藏在模块卡深处
- 共享入口层的展开区只能表达“补充概览”，不能用“查看全部待办”去误导用户；完整待办应继续交给通知中心或审批中心承接
- 共享入口层的顶部待办条目当前统一只允许进入中心页，不再直接跳聊天、协作空间、记录详情、验收页这类对象工作页
- PC Web 在共享入口层之下，还应补一层独立通知中心页，负责按分组、列表与右侧摘要继续承接高优先级事项；全局入口保持干净路径，通知项跳转再附带精确上下文
- 企业端在独立通知中心之外，还应补独立审批中心页，专门承接“需要我立即决策”的事项；审批中心与通知中心之间保持互跳，但不混用定位
- 从工作台顶部待办进入通知中心或审批中心时，应继续透传分组语义，例如 `group=confirmations / changes / reviews`，让左侧分组默认落到对应类别
- 从工作台顶部待办或模块弹层进入通知中心 / 审批中心时，应继续透传对象级锚点，例如 `itemId / approvalId / taskId / room / recordId`，让中心页不只落到正确分组，还能直接选中正确事项
- 审批中心第一批应围绕 `Approval` 对象渲染 `list / detail / action` 主链：列表负责选中对象，右侧详情负责解释当前审批背景，审批主动作只跟随 `approval.status + pendingAudience + decisionActions`
- 审批中心当前只承接 `confirmations / changes / reviews / cancellations` 4 组审批对象，不再把 `matching / followup` 混入审批中心列表
- PC 首页上的共享待办入口、模块弹层入口和首页快捷入口，下一阶段应统一只进入通知中心或审批中心，不再从首页直接深链聊天、协作空间、记录详情或验收页；对象页跳转改由中心页完成
- 从通知中心或审批中心进入聊天、协作空间、记录详情、验收页时，也应继续透传 `group + itemId / taskId / room / recordId / source`；目标页点击“返回来源”后，应回到原中心页并恢复原分组与原事项，而不是只回到默认列表；`center -> chat/workspace/record/acceptance -> return` 当前已按这条原则收口
- PC 通知中心 / 审批中心的右侧当前对象摘要区，应补一条轻量对象动作带，统一承接 `去聊天 / 去协作空间 / 看记录详情 / 去验收页` 这类对象动作；它只跟随当前 `selectedItem`，不替代顶部中心级 CTA
- 通知中心与审批中心右侧的对象动作带，应继续按共享对象动作合同渲染：主动作、次动作与对象页动作保持同一位置和顺序，不再按页面各自分叉
- 通知中心与审批中心右侧“当前对象”还应明确区分稳定态与失效态：稳定态展示对象摘要与动作带，失效态保留原分组与来源语义，只显示重选提示与安全动作，不能静默切到其他事项
- 通知中心与审批中心在当前对象失效态下，右侧只允许展示“查看当前分组首项”“清除当前定位”“查看全部事项”等安全恢复动作，不能默认替换成会把用户带去无关事项的常规主动作
- 通知中心与审批中心的顶部摘要卡、右侧空态和底部 footer 在 `invalid` 态下也必须复用同一组恢复动作；不能出现右侧已进入恢复态、但顶部或底部还在保留“去处理 / 去协作空间 / 去通知中心”这类对象级 CTA 的错位
- 中心页一旦执行恢复动作，后续聊天 / 协作空间 / 记录详情 / 验收页都必须以恢复后的当前对象为准；旧失效对象只能停留在历史来源里，不能继续作为当前对象下传
- 记录详情、协作空间、验收页必须遵循同一回链顺序：先回 immediate source，再由上一层按 `originSource` 回通知中心或审批中心；任何一页都不能直接越级回中心或静默改绑到其他对象
- 记录详情、协作空间、验收页的返回按钮文案也应尽量贴合真实目标，例如“返回聊天”“返回协作空间”“返回记录详情”“返回通知中心”“返回审批中心”，不要长期停留在泛化的“返回来源 / 返回列表”
- 详情页：对象详情或复盘详情，第一屏先给结论、关键指标与动作，随后承接时间线、资产和上下文
- 聊天页：左会话 / 中消息 / 右上下文三栏
- PC 聊天页头部必须承接轻量来源提示、返回来源入口和当前任务摘要；来源提示只做短标签，不膨胀成长说明区
- 协作空间：左任务树 / 中主区 / 右详情
- 协作空间的 PC 落地可接受“左任务切换器 / 中主工作区 / 右固定上下文”的稳定桌面工作区，不要求继续沿用过渡态长单列页面
- 当记录详情、协作空间、验收页形成二跳或多跳工作链时，导航上下文应拆成 `source + originSource`：`source` 表示直接上一页，`originSource` 表示最终通知中心或审批中心；对象级锚点继续保留 `itemId / group / taskId / room / recordId / nodeId / tab`
- 工作台、聊天、记录列表、记录详情、协作空间、验收页之间必须保留页面级上下文：`itemId / group / taskId / room / recordId / nodeId / tab / source / originSource`
- 页面级上下文只在“同一条工作链”内保留；用户手动切换任务或会话后，应主动清掉旧 `itemId / group / source / originSource / nodeId / tab / room / recordId`，避免把上一条工作链的来源和节点错带到当前对象
- 模块弹层里的 secondary 入口只允许保留创建页或泛列表页，不应再从首页直接保留聊天 / 协作 / 记录详情这类对象页直达
- 后台：列表优先，抽屉与右侧操作栏承接编辑
- 若短期内未完全做到三栏，也必须优先向“左导航 + 顶部状态 + 主工作区”收敛

### 4.3 禁忌

- 不要沿用 H5 底部 tabbar 心智
- 不要把所有东西做成大卡片流
- 不要让 PC 首页只剩营销式摘要和弹窗入口
- 不要让 PC 首页同时承担中心页和对象页两层入口职责
- 不要进入聊天后丢失原始来源链
- 不要只显示房间名而不显示当前任务语境
- 不要强制用户退回聊天列表，而忽略原来源页

## 5. 双端差异原则

- H5：单任务注意力
- PC：多任务并行处理
- H5：底部动作与 sheet
- PC：侧栏、抽屉、批量处理
- H5：最少字段
- PC：更高信息密度

## 6. 当前设计基线

后续所有页面走查，统一按本文件判断：

- 是否符合端定位
- 是否存在跨端错位
- 是否出现说明压过操作
- 是否出现“展示化过重、办公效率过低”的问题

当前阶段额外约束：

- H5 已允许使用独立 App 壳和底部导航，不再要求与 PC 共用页面壳层
- PC 已开始使用独立桌面业务壳，官网顶栏仅保留在门户页
- H5 工作台与协作页允许保留轻摘要，但必须把重历史与重详情下沉
- H5 企业端 / 人才端工作台第一阶段当前统一收成“优先处理 + 两个主入口 + 更多信息”，模块卡式首页不再作为默认首屏
- H5 协作空间首屏只保留当前任务摘要、一个主动作、一个次动作和轻量入口行，完整节点列表、最近记录、任务详情与重表单下沉到次级层
- H5 记录详情第三阶段当前统一收成“结论 + 动作 + 当前状态 + 附件留痕”，补充信息、交付范围、最新留痕、确认历史和过程节点全部下沉到折叠层
- H5 验收页第三阶段当前统一收成“当前结论 + 当前唯一待处理动作 + 动作结果”，评级依据、评分历史、信用影响和过程节点全部下沉到折叠层
- H5 聊天详情第四阶段当前已去掉重复摘要层，首屏只保留会话头、压缩后的任务确认卡、消息线程和输入区；版本历史与更长说明统一下沉到 sheet
- H5 聊天详情第五阶段当前已把消息展示、附件展示、房间预选和刷新判断移出页面主文件，页面优先只保留会话编排、主动作和对象级 UI
- PC 列表页已进入办公端收口阶段，企业端人才广场、人才端任务广场、发单记录、接单记录和后台列表统一优先使用行式列表 / 表格 + 筛选 + 批量动作 + 详情承接的结构，卡片瀑布只允许作为过渡态
