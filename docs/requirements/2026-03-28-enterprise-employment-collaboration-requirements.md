# 企业雇佣协作结算平台需求文档

> 版本：2026-03-28  
> 事实源说明：本文件基于“规范文件夹”中的目标态文档整理，作为当前根 `docs/` 下新的需求事实源。`规范文件夹` 保留为参考基线，不直接参与日常维护。

## 1. 文档目标

定义平台的业务目标、角色边界、核心流程、功能范围与验收标准，用于统一产品、设计、研发、测试、运营和交付口径。

本期平台不再只定义为“AI 人才协作市场 MVP”，而是明确为：

- 招聘撮合
- 在线协作
- 审批与留痕
- 验收与结算
- 信用沉淀与复购

的一体化雇佣协作平台。

## 2. 产品定位

平台服务的不是一次性发帖招聘，而是“从找人到把事做完并完成结算”的完整闭环。

平台同时满足两类心智：

- 类 Boss 直聘：快速找人、即时沟通、短链路确认合作
- 类 OA 办公：任务推进、文件流转、审批留痕、验收结算、财务闭环

核心价值：

- 企业更快找到可交付的人
- 人才更清楚地接单、协作、收款、沉淀信用
- 平台更容易建立可信交易与履约中台

## 3. 角色定义

### 3.1 企业端

典型角色：

- 企业主
- 招聘负责人
- 项目负责人
- 业务成员

核心诉求：

- 快速注册和主体认证
- 发布职位或任务
- 筛选、邀约、确认人才
- 在线沟通与变更确认
- 推进协作、审批交付、发起验收和结算

### 3.2 人才端

典型角色：

- 自由职业者
- 外包成员
- 兼职人才
- 独立开发者 / 设计师 / 内容创作者

核心诉求：

- 完善实名、专长、作品和档期
- 浏览任务、接单、确认需求
- 提交进度、附件和请款材料
- 完成验收、收款、沉淀评价与信用

### 3.3 平台侧

包含：

- 运营
- 财务
- 审核
- 风控

核心诉求：

- 审核主体与资料
- 处理争议、异常与风控
- 承接结算、发票、对账
- 管理配置、推荐与治理动作

## 4. 双端定位

### 4.1 H5 仿 App

H5 负责高频、轻操作、强提醒、强触达场景。

适合：

- 注册登录
- 登录后的待办处理与个人入口
- 消息与聊天
- 任务确认
- 进度提交
- 轻量审批
- 验收确认
- 评分与个人中心

不适合：

- 大表格批量处理
- 多窗口并行对比
- 复杂财务与权限配置
- 长时间停留在高信息密度的桌面驾驶舱页面

### 4.2 PC Web

PC 负责重操作、复杂管理、多任务并行场景。

适合：

- 企业工作台
- 企业 / 人才桌面驾驶舱
- 多任务总览
- 人才筛选与批量管理
- 协作工作区
- 审批中心
- 财务结算中心
- 后台审核与风控

## 5. 核心业务流程

### 5.1 企业端主流程

1. 注册并完成主体认证
2. 发布职位或任务
3. 系统基于 AI / 规则拆解需求
4. 推荐候选人才并支持手动筛选
5. 发起聊天并确认合作
6. 进入协作工作区
7. 过程内处理进度、文件、审批、变更
8. 发起验收、结算、付款与开票
9. 完成评分与结案
10. 进入复购与复用阶段

补充说明：

- H5 登录后优先进入轻量工作台，只暴露待办、聊天、发布和协作入口
- PC 登录后优先进入桌面工作台，承接待办、风险、记录与多任务入口
- 企业协作空间在 H5 上只保留当前任务、关键节点与主动作；PC 上承接多任务总览与上下文查看

### 5.2 人才端主流程

1. 注册并完成实名认证
2. 完善资料、技能、作品和档期
3. 浏览任务或接受邀约
4. 通过聊天确认任务
5. 进入协作与交付
6. 提交进度、附件、请款或补充材料
7. 确认验收与结算
8. 对企业评分
9. 沉淀收入、作品和信用画像

补充说明：

- H5 人才端首页只放待确认、聊天、任务广场和协作入口，不承接重历史信息
- 人才端在确认任务前，必须在聊天或任务详情中看到预算、工期、交付与协作安排
- 人才端协作空间在 H5 上保持单任务执行台，PC 才承接更高密度的多栏办公视图

### 5.3 关键过程约束

- 所有关键状态必须可见、可回看、可追踪
- 需求变更必须有版本和处理记录
- 取消任务必须双方确认
- 提前完成必须经过 AI 审核与对方确认
- 评级与结算必须有明确规则

## 6. 功能范围

### 6.1 前台

- 官网 / 落地页
- 注册登录
- 企业入驻
- 人才入驻
- 工作台
- 人才广场 / 任务广场
- 人才详情 / 任务详情
- 聊天
- 协作工作区
- 验收与评级
- 发单记录 / 接单记录
- 我的页面

前台端内分工补充：

- H5：首页入口、登录注册、工作台、聊天、协作、记录、我的
- PC：首页入口、桌面工作台、聊天工作区、协作工作区、广场与记录列表
- PC 的广场页与记录页默认应是摘要 / 筛选 / 列表 / 详情承接的办公页，不应退化成纯展示卡片流
- 从记录详情进入 PC 聊天后，聊天右侧应继续承接同一条记录的对象摘要：结论、阶段、金额、评级、最近留痕与返回记录详情入口，而不是退回纯任务摘要

### 6.2 后台

- 审核管理
- 风控管理
- 用户管理
- 任务管理
- 结算与发票管理

### 6.3 平台能力

- AI 拆解与建议
- 聊天与通知
- 文件与附件
- 审批流
- 结算与发票
- 信用评分与复购沉淀

## 7. 结算与评级规则

- 企业可在验收阶段给出 `S / A / B` 评级
- 对应结算比例：
  - `S`：100%
  - `A`：80%
  - `B`：30%
- `S` 级需要有更强的仪式化结果展示
- 人才也可对企业评分
- 企业评分会影响首页案例、工作台摘要与企业信用展示

## 8. 非功能要求

- 关键状态变化需要有通知与提醒
- 关键写操作必须服务端校验
- 文件、支付、发票、审计不能只依赖前端状态
- H5 与 PC 共享业务事实，但交互形态必须分端处理

## 10. 验收标准

### 产品层

- 企业能从注册走到发任务、选人、协作、验收、结算
- 人才能从注册走到接单、协作、交付、收款、评分
- H5 与 PC 都能覆盖各自高频主路径

### 交互层

- 用户始终知道当前状态与下一步动作
- H5 不出现 PC 缩放式双栏工作台
- PC 不退化成纯展示卡片流
- H5 工作台首屏只保留待办、状态与主入口
- H5 聊天必须保持“列表页 / 详情页”拆分
- PC 聊天应优先表现为“左会话 / 中消息 / 右上下文”

### 业务层

- 任务、变更、验收、取消、评分都可回看
- 结算比例、评级规则、状态流转明确可验证
