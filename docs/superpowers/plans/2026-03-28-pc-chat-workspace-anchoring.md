# 2026-03-28 PC 聊天与协作空间双向锚点

## 目标

把 PC Web 聊天页与协作空间之间的往返路径收成稳定双向锚点：用户从聊天进入协作空间时能落到正确任务和正确上下文，从协作空间回到聊天时也能继续回到当前会话，而不是退回泛列表。

## 本阶段范围

- [x] 聊天页进入协作空间时保留当前任务、房间与记录上下文
- [x] 协作空间进入聊天时保留当前任务、房间与来源链
- [x] 协作空间页补当前任务 / 当前节点与聊天入口的精确锚点
- [x] 根 `docs/` 与四个子项目副本文档同步本阶段结果
- [x] 运行 `frontend`、`frontend_mobile_h5`、`backend` 验证

## 设计方向

- 聊天与协作空间是同一条工作链的两个工作面，不应该像两个孤岛页面
- 返回链优先回到“当前会话 / 当前任务”，不退回大列表
- 锚点提示保持轻量，只提示当前任务与当前节点，不堆大说明区

## 后续阶段

- PC 记录详情与聊天共享右侧上下文
- 批量通知 / 审批处理与聊天详情联动
- 审批中心与协作空间的状态级联更新

## 本阶段结果

- `MessagesPage` 与 `WorkspacePage` 之间的双向入口已经补齐 `taskId / room / recordId / nodeId / source` 级别的上下文链，不再只依赖 `room`
- 聊天页当前已支持“仅有 `taskId` 时反查任务房间”的兜底，避免从协作空间或记录详情返回聊天时直接掉回最近会话
- 协作空间当前任务 / 当前节点不再只停留在 URL；节点列表会形成当前态，节点弹窗与 URL 保持双向同步
- 协作空间基础竞态已补第一层防护：旧请求不会再轻易覆盖新任务页面；手动切任务时会清掉旧 `room / recordId / nodeId / source`
