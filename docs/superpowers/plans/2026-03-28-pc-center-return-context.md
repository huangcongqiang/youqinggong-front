# 2026-03-28 PC 通知 / 审批返回链对象级保持

## 目标

让通知中心 / 审批中心进入聊天、协作空间、记录详情后，不只保留来源页标记，还能继续保留对象级锚点，并在“返回来源”时恢复原分组与原事项。

## 本阶段范围

- [x] 通知中心进入聊天 / 协作 / 记录详情时继续附带 `group + itemId + taskId / room / recordId`
- [x] 审批中心进入聊天 / 协作 / 记录详情时继续附带 `group + itemId + taskId / room / recordId`
- [x] 聊天页继续保留 `itemId / group`，并在返回审批中心 / 通知中心时回传
- [x] 协作空间继续保留 `itemId / group`，并在返回审批中心 / 通知中心时回传
- [x] 记录详情继续保留 `itemId / group`，并在返回审批中心 / 通知中心时回传
- [x] 根 `docs/` 与四个子项目副本文档同步本阶段结果
- [x] 运行 `frontend`、`frontend_mobile_h5`、`backend` 验证

## 设计方向

- `source` 只表达“来自哪里”，`itemId / taskId / room / recordId` 才表达“来自哪一条对象”
- 直接进入聊天、协作空间、记录详情时，目标页必须继续保留对象级锚点
- 点击“返回来源”时，应优先恢复原对象；只有对象失效时才退回分组或列表

## 后续阶段

- 多跳返回链继续升级成更稳定的来源栈
- 审批中心 / 通知中心与桌面壳全局提醒联动
- 审批、请款、开票、对账、结算正式链路

## 本阶段结果

- 通知中心与审批中心当前会继续把对象级锚点带到聊天、协作空间和记录详情
- 聊天、协作空间和记录详情当前会继续保留 `itemId / group`，回通知中心 / 审批中心时不再只靠空 query 返回
- 当前返回链已从“只认来源页”升级到“来源页 + 对象锚点”，后续只剩多跳来源栈的进一步正式化
