# 2026-03-28 PC 首页只进中心页

## 目标

把 PC 首页上的共享待办入口、模块弹层入口和桌面壳首页快捷入口统一收口为“先进入通知中心或审批中心”，不再从首页直接深链聊天、协作空间、记录详情或验收页；对象页跳转改由中心页在完成分组与对象预选后再发起。

## 本阶段范围

- [ ] 首页共享待办入口只进入通知中心或审批中心
- [ ] 首页模块弹层与首页快捷入口不再直接跳聊天 / 协作空间 / 记录详情 / 验收页
- [ ] 首页进入中心页时继续透传 `group + itemId / taskId / room / recordId`
- [ ] 中心页成为对象级预选与二次跳转的唯一入口层
- [ ] 根 `docs/` 同步“首页 -> 中心页 -> 对象页”的新职责边界
- [ ] 补一组新的文档验收口径，替换“首页直接进对象页”的旧表述

## 设计方向

- 首页只负责“把用户送到正确中心页”，不负责“把用户直接送进具体对象页”
- 首页仍可继续透传 `group + itemId / taskId / room / recordId`，但这些锚点只用于中心页预选，不再作为首页直接跳对象页的依据
- 企业端首页默认优先进审批中心，人才端首页默认优先进通知中心
- 通知中心 / 审批中心负责三件事：
  - 按 `group` 落到正确分组
  - 按 `itemId -> room -> recordId -> taskId` 预选正确事项
  - 从当前事项再进入聊天、协作空间、记录详情、验收页
- 对象页的来源链保持不变，仍按 `source + originSource + itemId / group / taskId / room / recordId` 继续回到中心页

## 不在本阶段

- 不调整中心页进入对象页后的返回链规则
- 不改对象页内部的 `source + originSource` 结构
- 不处理 H5 首页入口策略

## 验证矩阵

- 企业首页顶部待办主动作进入审批中心，而不是直接进入聊天 / 协作空间 / 记录详情 / 验收页
- 人才首页顶部待办主动作进入通知中心，而不是直接进入聊天 / 协作空间 / 记录详情 / 验收页
- 首页模块弹层进入中心页时，`group + itemId / taskId / room / recordId` 仍能让中心页准确预选对象
- 中心页在预选后继续进入聊天、协作空间、记录详情、验收页时，返回链仍能恢复原中心分组与原事项

## 预期结果

- 首页入口职责更单一，避免“首页既像驾驶舱又像对象路由器”
- 中心页成为稳定的桌面入口层，首页和对象页之间不再直接耦合
- 后续批量处理、已读状态、跨中心互跳和来源链维护都能继续收敛到中心页处理
