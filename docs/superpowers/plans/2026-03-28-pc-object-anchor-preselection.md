# 2026-03-28 PC 通知 / 审批对象级预选

## 目标

把 PC Web 企业端工作台进入审批中心、人才端工作台进入通知中心的链路，从“只落到某个分组”升级成“直接落到对应对象”，减少二次寻找和误选。

## 本阶段范围

- [x] 企业端工作台进入审批中心时补齐 `itemId / taskId / room / recordId` 锚点
- [x] 人才端工作台进入通知中心时补齐 `itemId / taskId / room / recordId` 锚点
- [x] 企业端模块弹层进入审批中心时按模块挑选首选事项，而不是一律落到第一个待办
- [x] 人才端模块弹层进入通知中心时按模块挑选首选事项，而不是一律落到第一个待办
- [x] 审批中心消费对象级 query，并自动选中对应事项
- [x] 通知中心消费对象级 query，并自动选中对应事项
- [x] 根 `docs/` 与四个子项目副本文档同步本阶段结果
- [x] 运行 `frontend`、`frontend_mobile_h5`、`backend` 验证

## 设计方向

- 工作台进入中心页时，不只传 `group`，还要把对象级锚点一起带过去
- `itemId` 是最直接的选中锚点，`taskId / room / recordId` 作为回退匹配键
- 模块弹层不应无差别指向“第一个待办”，而应根据聊天、协作、记录、发布等模块语义挑选首选事项
- 中心页选中某一条事项后，URL 也应同步到对应锚点，保证刷新后仍可恢复

## 后续阶段

- PC 通知中心 / 审批中心与聊天、协作空间之间的返回链继续对象级保持
- PC 审批、请款、开票、对账、结算链路
- 通知 / 审批与桌面壳全局提醒联动

## 本阶段结果

- 企业端工作台进入审批中心时，现在会附带 `group + itemId + taskId/room/recordId`
- 人才端工作台进入通知中心时，现在会附带 `group + itemId + taskId/room/recordId`
- 审批中心与通知中心会优先按对象级锚点匹配事项，必要时自动切到正确分组并选中对应条目
- 模块弹层进入中心页时，已经开始按“聊天 / 协作 / 记录 / 发布”语义选择更合适的首选事项
