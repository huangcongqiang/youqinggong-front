# 2026-03-28 PC 首页待办中心入口合同

## 目标

把 PC Web 工作台顶部共享待办入口，统一收成“首页只进中心页，中心页再进对象页”，避免首页自己直冲聊天、协作空间、记录详情等对象页。

## 本阶段范围

- [x] 企业端顶部待办条目统一先进入审批中心
- [x] 人才端顶部待办条目统一先进入通知中心
- [x] 企业端 / 人才端模块弹层主动作统一先进入中心页
- [x] 模块弹层里的“保留原入口”只保留给泛页面入口，不再给聊天 / 协作 / 记录详情这类对象页
- [x] 新增共享 route/anchor helper，统一中心入口与模块首选事项选择逻辑
- [x] 根 `docs/` 与四个子项目副本文档同步本阶段结果
- [x] 运行前端验证

## 设计方向

- 首页的共享待办层只负责“把人送进正确的中心页”
- 中心页才负责“把人落到正确的对象页”
- 允许保留的直接入口，只能是泛页面或创建页，例如发布任务、任务广场
- 聊天、协作空间、记录详情、验收页都属于对象工作页，不应再由首页待办条目直接跳入

## 本阶段结果

- 企业端共享待办 4 条当前统一进入审批中心，并继续透传 `group + itemId + taskId / room / recordId`
- 人才端共享待办 4 条当前统一进入通知中心，并继续透传 `group + itemId + taskId / room / recordId`
- 新增 `frontend/src/utils/attentionNavigation.js`，统一承接中心入口 route 构建与模块语义选项选择
- 模块弹层里直达对象页的 secondary 入口已收掉，只给发布任务 / 任务广场这类泛入口保留直达

## 后续阶段

- 中心页补更完整的当前对象动作带，例如去聊天、去协作空间、查看记录详情、去验收页
- `DesktopAttentionHub` 继续去路由语义，向更纯的展示组件收口
- 桌面壳里的待办、通知、审批继续统一成更稳定的工作入口层
