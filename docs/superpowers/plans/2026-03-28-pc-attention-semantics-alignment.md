# 2026-03-28 PC 待办入口语义收敛

## 目标

把 PC Web 企业端 / 人才端工作台顶部的共享待办入口层收得更诚实、更稳定：统一优先消费通知数据契约、补分组深链、避免“查看全部待办”这类误导性表述，并让审批中心 / 通知中心能按分组正确落位。

## 本阶段范围

- [x] 收敛 `DesktopAttentionHub` 的入口契约与展开语义
- [x] 企业端工作台改为优先消费 `notificationItems / notificationGroups`
- [x] 人才端工作台改为优先消费 `notificationItems / notificationGroups`
- [x] 审批中心 / 通知中心支持通过 `group` query 预选分组
- [x] 根 `docs/` 与四个子项目副本文档同步本阶段结果
- [x] 运行 `frontend`、`frontend_mobile_h5`、`backend` 验证

## 设计方向

- 工作台顶部待办入口层优先承接“高优先级入口”，不是“完整待办列表页”
- 展开区只能表达“补充概览”，不能伪装成“查看全部待办”
- 工作台进入审批中心 / 通知中心时，应继续带上分组语义，减少二次筛选
- 缺少处理入口的数据项不能静默跳默认页，避免把数据缺口伪装成交互成功

## 后续阶段

- PC 通知中心与审批中心的对象级预选（`itemId / recordId / taskId`）
- 审批、请款、开票、对账、结算链路
- 审批中心与协作空间的状态级联更新

## 本阶段结果

- 企业端 / 人才端工作台顶部共享待办入口层现在优先消费 `notificationItems / notificationGroups`，只有缺失时才回退到旧的 `attentionItems`
- 工作台顶部展开区统一改为“补充概览”，不再误导成“查看全部待办”
- 工作台进入企业端审批中心与人才端通知中心时，会附带 `group` query，让左侧分组默认落到对应高优先级类别
- `DesktopAttentionHub` 的单项入口不再通过默认 fallback 静默跳页；缺失处理入口时会保持禁用展示态
