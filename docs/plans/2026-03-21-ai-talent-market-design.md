## 1. 项目目标

打造一款围绕 AI 协作场景的“人才市场”平台，连接企业端与人才端，让任务发布、AI 需求拆解、人才匹配、协作沟通、进度审查、验收评价形成闭环。

本期实现目标为 MVP：

- 提供 H5 + PC 自适应的前台 Vue 应用
- 提供 PC 端 Vue 管理后台
- 提供 Java API 骨架和 SQL 数据模型
- 将企业入驻、个人虚拟企业申请、AI 拆单、任务匹配、协作验收、评价、后台管理串成可演示流程

## 2. 角色与核心流程

### 企业端角色

- 企业入驻：上传营业执照、品牌材料、联系人信息
- 个人入驻：申请虚拟企业，上传身份证和实名信息
- 发起任务：支持语音输入、文本输入
- AI 拆解：自动输出任务模块、建议工期、技能标签、风险提醒
- 人工确认：企业端确认 AI 输出后进入匹配
- 匹配人才：系统先推荐 4 位候选人，企业端可直接选择，也可进入人才广场补充筛选
- 协同交付：与人才确认需求、排期、里程碑
- 验收与评价：工作完成后验收并评分

### 人才端角色

- 人才入驻：填写简介、技能、作品、服务方向、接单日历
- 查看任务：主动浏览任务列表或等待推荐
- 接受任务：就需求和工期进行确认
- 上传进度：提交图文、视频、代码、资料
- AI 审查：接收 AI 对进度与完成度的反馈建议
- 消息沟通：与企业端在消息中心沟通
- 评价与结算：完成后对企业端评分，并查看收入与作品统计

### 管理后台

- 审核企业与个人入驻材料
- 管理用户、任务、进度、消息、评价
- 手工创建任务和用户
- 查看平台经营看板与风险提醒

## 3. 方案比较

### 方案 A：模块化单体 + 双前端应用（推荐）

- `frontend`：Vue 3，覆盖官网 + 企业端 / 人才端工作台，兼容 H5/PC
- `admin`：Vue 3，PC 管理后台
- `backend`：Java 模块化单体 API，统一鉴权、任务、聊天、AI 编排、审核
- `sql`：MySQL 8 关系模型，后续可接对象存储与消息队列

优点：

- MVP 交付速度快
- 业务模型集中，便于快速迭代
- 后续可平滑拆出 AI 服务、聊天服务、审核服务

缺点：

- 单体服务后期需要按领域拆分

### 方案 B：前后端分离 + 纯原型 Mock

- 前端快速出效果
- 后端只保留静态 JSON 或 API 文档

优点：

- 开发速度最快

缺点：

- 无法验证实际接口结构与数据闭环
- 后续重构成本更高

### 方案 C：微服务起步

- 用户中心、任务中心、消息中心、AI 服务、审核服务全部独立

优点：

- 理论扩展性强

缺点：

- 当前阶段明显过重
- 运维、联调、排障成本高

## 4. 推荐结论

采用方案 A：模块化单体 + 双前端应用。

原因：

- 你的需求包含多角色、多状态、多附件、多 AI 节点，非常适合先用统一的数据模型跑通闭环
- 现阶段需要尽快把“产品感”和“可演示能力”做出来，而不是先投入在分布式治理
- 后续当任务量、消息量、文件量增长时，再按领域边界拆服务即可

## 5. MVP 功能范围

### 前台 Web（H5/PC）

- 首页：品牌价值、平台流程、角色入口、AI 能力展示
- 企业端工作台：
  - 入驻审核状态
  - AI 任务发布器
  - 任务拆解确认
  - 人才推荐列表
  - 协作工作区
- 人才端工作台：
  - 个人资料与作品
  - 任务广场
  - 日历接单
  - 任务管理
  - 消息中心
  - 收入与评价
- 协作工作区：
  - 需求确认
  - 工期确认
  - 里程碑跟踪
  - 进度上传
  - AI 审查建议
  - 验收与评分

### 管理后台

- 经营看板
- 用户管理
- 入驻审核
- 任务管理
- 风险提醒
- 手工创建用户/任务

### Java API

- 首页/看板数据接口
- 企业端数据接口
- 人才端数据接口
- 管理台数据接口
- AI 任务拆解接口（MVP 先返回模拟结构）
- 腾讯 IM 配置与任务会话映射预留接口

- `/` 管理驾驶舱
- `/users` 用户管理
- `/tasks` 任务管理

## 7. 数据模型

核心实体：

- `users`：平台用户，支持 `ADMIN / BUSINESS / TALENT`
- `organizations`：企业与虚拟企业信息
- `organization_verifications`：企业材料审核记录
- `talent_profiles`：人才资料、技能、作品、评价统计
- `tasks`：任务主表
- `task_ai_analyses`：AI 拆解结果
- `task_assignments`：候选人推荐与确认
- `task_negotiations`：工期与需求确认记录
- `task_milestones`：里程碑
- `task_progress_updates`：进展反馈
- `task_files`：上传附件
- `chat_conversations` / `chat_messages`：消息系统
- `reviews`：互评
- `calendar_availability`：接单日历

## 9. 非功能要求

- 性能：
  - 首屏加载目标 < 2.5s
  - API p95 < 300ms（MVP）
- 安全：
  - 入驻证件、身份证等敏感信息仅后台可见
  - 上传材料必须走鉴权和审计
- 可维护性：
  - 前端按页面与组件分层
  - 后端按领域模块拆分目录
- 扩展性：
  - AI 服务接口化，后续可接大模型厂商
  - 聊天能力与平台状态解耦，二期可切换腾讯 IM 实时通信
- 兼容性：
  - Frontend 同时兼容移动端与桌面端
  - 管理后台以桌面端宽屏为主

## 10. 分阶段实施计划

### Phase 1：MVP 骨架

- 完成双前端应用结构
- 搭建 Java API 和 mock 数据
- 输出 SQL 表结构
- 跑通角色页面和任务闭环展示

### Phase 2：真实业务接入

- 接入登录注册、短信/邮箱
- 接入企业认证与实名认证
- 接入文件上传与对象存储
- 接入真实 AI 拆解与 AI 审查

### Phase 3：交易与协作增强

- 接入支付、结算、发票
- 增加 IM 实时通信
- 增加推荐算法与人才画像
- 增加 SLA、投诉仲裁、风控

## 11. 当前实现边界

本次交付实现的是高保真 MVP 骨架：

- 界面完整、结构可运行、交互可演示
- Java API 可启动并返回核心 JSON 数据
- SQL 结构可直接作为后续数据库设计基础
- AI 能力以 mock 数据方式预留真实接入点

后续若继续开发，可在此基础上逐步替换 mock 数据与模拟 AI 逻辑，并接入腾讯 IM 的真实会话能力。
